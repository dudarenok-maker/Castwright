## What this covers

Streams chapter audio generation via Server-Sent-Events. The client opens a long-running POST with `{ modelKey, chapterIds?, force? }`; the server iterates chapters, synthesises each character's same-speaker groups via the chosen TTS provider, emits ticks for progress / assembling / completion / failure / idle, and on `chapter_complete` encodes the concatenated PCM to MP3 (LAME VBR V2 via system ffmpeg — see [plan 28](28-chapter-audio-format.md) for the format contract) and writes `<slug>.mp3` + `<slug>.segments.json`. The client maps ticks into per-chapter/per-character state transitions in the chapters slice, plays the resulting MP3 via the MiniPlayer, and forwards a `pendingRegen` spec on the next stream open when the user regenerates.

## Invariants to preserve

- `realStreamGeneration` returns a canceller function that calls `controller.abort()`; aborts surface as DOMException `AbortError` and are NOT mapped to a failure tick. Treat abort as "the canceller did its job."
- **Every live broadcast tick carries run-level aggregates** (`runDone`,
  `runTotal`, `runInProgress`) so the global header `GenerationPill`
  keeps moving accurately even when the user has navigated away from
  the generating book (Bug E). The server's `broadcast()` wrapper
  (`server/src/routes/generation.ts:97`) injects them into every
  emitted event from the job's own counters; `RunningJob` seeds
  `runDoneBase` from chapters with audio on disk that aren't in this
  run's scope, and the loop increments `completedThisRun` /
  `runInProgress` as chapters advance. Catch-up replay ticks
  (`chapter_complete` for chapters already on disk before the run
  started) do NOT carry aggregates — they're emitted via the
  subscriber's direct `send`, not through `broadcast`, because they
  pre-date the job.
- **Cross-book heartbeat is driven by the tick payload, not the slice**
  (Bug E). The `chapters/applyGenerationTick` reducer's cross-book
  guard (`src/store/chapters-slice.ts:267`) correctly skips per-row
  mutation when `currentBookId !== activeStream.bookId`, but that
  also skipped the `lastTickAt` update. The middleware's post-tick
  block (`src/store/generation-stream-middleware.ts:385-410`) now
  forks: slice-matches-handle takes `setActiveStream(snapshotFromChapters(...))`
  as before; cross-book takes `updateActiveStreamProgress` reading
  `runDone` / `runTotal` / `runInProgress` from the tick payload. The
  reducer always bumps `lastTickAt`; counters are overwritten only
  when the payload carries them, so an older-server SSE without
  run* fields still keeps the pill alive (no spurious "Stalled"
  after STALL_THRESHOLD_MS).
- `GenerationTick` union: `'progress' | 'chapter_assembling' | 'chapter_complete' | 'chapter_failed' | 'idle'`. Payload fields per type:
  - `progress` → `chapterId, characterId, progress, currentLine, totalLines`. One emitted per same-speaker group; `characterId` is the live speaker.
  - `chapter_assembling` → `chapterId` (required); `totalGroups?`, `durationSec?` carry the run summary about to land on disk. Fired between the last group and the MP3 encode + write (encoder + atomic-rename detail in [plan 28](28-chapter-audio-format.md)).
  - `chapter_complete` → `chapterId`; `totalLines?`, `audioModelKey?`, `durationSec?` optional. `durationSec` is a belt-and-suspenders duplicate of the `chapter_assembling` carry — emitted on every completion (run-time and replay) so the chapter row's duration label flips to the real audio length even when the assembling tick was lost on the wire (hidden tab, cross-book guard, parallel-chapter coalesce). The reducer applies it idempotently: re-stamping the same value the assembling tick already wrote is a no-op.
  - `chapter_failed` → either `chapterId + errorReason` (per-chapter synth failure), OR **no `chapterId`** with `errorReason` (stream-level setup error: modelKey rejected, cast missing, sidecar down). The chapter-less form populates `chaptersState.lastError` and flips the currently in-flight chapter to `failed`.
  - `idle` → no extra fields. Fired when the server has nothing to do.
- SSE frame format mirrors analysis stream: `data: <json>\n\n`, `\n`-joined multi-line `data:`.
- `chapterIds?: number[]` — optional subset; when absent, server defaults to "all chapters lacking audio on disk."
- `force?: boolean` — when true, re-synthesises chapters even if a `.mp3` already exists for that chapter (`chapterAudioExists` in `server/src/workspace/chapter-audio-file.ts` probes only `.mp3`).
- Per-character state inside a chapter: `applyGenerationTick` keeps the active speaker as `in_progress` and any previously-active speaker is demoted back to `queued` (NOT `done`). `done` only lands on `chapter_complete`, which flips every non-skipped character at once. `skipped` is preserved. If the tick names a `characterId` not in this chapter's cast (or one marked `skipped`), per-character state is left untouched — without this gate the previous in-progress speaker would silently flip to `queued` with nobody taking their place, and the Generate view would lose its "who is speaking right now" signal. The previous design flipped the prior speaker to `done` on every speaker change, which was a lie when the character had more lines later in the chapter — by line 13 of an 82-line chapter every cast member had spoken once and the expanded row showed three full-green "Done" bars while 80 % of the work was still ahead. Real per-character completion is now derived in the Generate view from `chapter.currentLine` + manuscript line positions (`characterLinePositionsByChapter` + `linesDoneAt` in `src/lib/generation-progress.ts`), not from the slice's status field.
- **Per-character bar widths come from the manuscript, not the SSE status.** The expanded chapter row's per-character bar is filled to `linesDoneAt(positions, chapter.currentLine) / characterStats.lines` — the truthful fraction of this character's lines that have been synthesised. The slice's per-character status (`in_progress`/`queued`/`skipped`/`failed`) still drives the colour and label (Generating/Queued/Skipped/Failed/Done), but the _width_ is always real. When the chapter is `done` or the character's derived count meets their total, the bar pins to full green; pinning on the slice's `done` flag alone would either lie or lag depending on which side the discrepancy fell on.
- **Regenerating a rendered chapter must NOT trust a stale per-character `done`.** Completion of a single cast row is derived purely from `chapter.currentLine` + manuscript line positions via `characterRowProgress` (`src/lib/generation-progress.ts`); it honors the slice's per-character `status` only to read `skipped`, and treats the row as fully done only when the WHOLE chapter is `done` (`chapter.state === 'done'`, which `chapter_complete` sets atomically with every character's `'done'`) or the derived count meets the character's total. This is load-bearing on regenerate: re-streaming a chapter whose `.mp3` still exists lets `hydrateFromBookState` re-seed every cast member as `'done'` (`seedDone`), and `applyGenerationTick` only un-`done`s the live speaker — so without deriving from `currentLine` the expanded row showed each speaker pinned at a full-green "Done" until its first sentence of the new run (the counters never visibly reset to zero). Pinned by `src/lib/generation-progress.test.ts` ("the regenerate stale-Done regression") and `src/views/generation.test.tsx` ("resets per-character rows when regenerating, ignoring stale 'done' statuses").
- **`currentLine` resets on regenerate.** All three regenerate reducers (`regenerateChapter`, `regenerateCharacter`, `batchRegenerateCharacters`) set `currentLine: 0` on any target chapter that is being reset (always for `regenerateChapter` in scope; only when the chapter was `done` for the character-scoped variants). Without this, the gap between regenerate firing and the first fresh `progress` tick showed the pre-regenerate fractional progress on top of a now-Queued chapter — the "stitching failed → goes back to nothing" path looked confused because the bars hadn't actually gone back to zero.
- The next-chapter auto-promote is driven by the server's own `progress` tick for the next chapterId — the client does **not** auto-promote on `chapter_complete`.
- `Chapter.phase: 'assembling' | null` is a UI-only sub-state set on `chapter_assembling` and cleared on `chapter_complete` / `chapter_failed`. It lets the Generate view show a neutral striped bar + "Assembling…" pill instead of the synthesis gradient stuck near 99 %. Not part of the OpenAPI schema.
- `pendingRegen: { chapterIds, force: true } | null` is set by the three regenerate reducers (`regenerateChapter`, `regenerateCharacter`, `batchRegenerateCharacters`) and consumed by the Generate view, which forwards it to the next `streamGeneration` call **and immediately dispatches `consumePendingRegen` to drain it**. `regenEpoch` is a monotonic counter the view watches as a `useEffect` dep so a repeat regenerate (same chapter, same character) still re-opens the SSE. Draining the spec on open (rather than waiting for the server's `idle` tick) is required because Pause aborts the SSE before any `idle` tick reaches the client: if the spec stuck around, every Resume would re-forward `force:true` and wipe the in-flight chapter, looping forever. The `idle` tick still clears `pendingRegen` defensively (in case a run completes without a pause). `regenEpoch` is never reset.
- `chaptersState.generationStartedAt` is set on the first `progress` or `chapter_assembling` tick of a run and drives the elapsed-based ETA. Cleared on `idle` when no chapter is `in_progress` or `queued`.
- Malformed ticks are logged and skipped; do not throw.
- **Per-bookId server mutex.** `server/src/routes/generation.ts` keeps a module-level `Map<bookId, AbortController>`; when a new POST arrives for a book that already has an in-flight handler, the prior controller is aborted. `synthesiseChapter` checks the signal between groups (throws `AbortError`, caught silently by the route's chapter loop) and forwards it into each `provider.synthesize` call so the sidecar fetch is cancelled mid-flight rather than running the rest of a slow group to completion. `req.on('close')` also fires the same controller so a client abort (Pause, refresh, regenerate) stops the run within seconds instead of finishing the current chapter. The cleanup block only deletes the Map entry if the entry still points at _this_ controller — a newer handler that has already displaced us owns the slot.
- **Pause = newest-request-wins.** Pre-mutex behaviour was "let the in-flight chapter finish before stopping," which meant a pause→resume cycle (or a stacked regenerate during synthesis) could pile up parallel server-side handlers all hitting the single-threaded sidecar. With the mutex + AbortSignal threading, Pause aborts the current synth call between groups and the next POST starts cleanly. Resume after Pause picks up from disk state (chapters already on disk are skipped).
- **Overall-progress weighting is sentence-count based, not totalLines based.** `overallProgress` (`src/lib/generation-progress.ts`) weights each chapter by, in precedence order: (1) the manuscript sentence count from `state.manuscript.sentences`, (2) the live `totalLines` carried on the chapter, (3) the mean of whatever weights resolved above, (4) 1 (equal-weight) when nothing has resolved yet. The original `weightedNum / totalLinesSum` math dropped any chapter without `totalLines` to weight 0 — so 3 hydrated-Done chapters + 1 in-flight chapter at 4 % rendered as "4 %" overall instead of ~75 %. The manuscript count is the canonical book-shape weight and is available as soon as analysis finishes, so it short-circuits the totalLines-or-fallback path entirely on the happy path.
- **Per-chapter and per-character readouts come from the manuscript, not the SSE.** `characterStatsByChapter` (same file) groups sentences by `chapterId → characterId → { lines, words }`; the Generate view renders a `<words> · <lines> · <speakers>` subtitle under each chapter title and a `<lines> · <words>` chip next to each character name in the expanded row. Word count strips `[audio-tags]` before counting tokens so a `[laughs]` chip doesn't inflate the figure. These render even before generation starts (queued state) because they're a property of the analysed manuscript, not the synthesis stream.
- **Per-speaker voice routing uses the full cast.json hint, not just the character id.** `synthesiseChapter` builds a `CharacterHint` (`description`, `role`, `gender`, `ageRange`, `tone`, `evidence`) for every group and passes it to `pickVoiceForEngine` — same shape the voice-sample preview uses. Skipping any of these fields collapses non-narrator speakers onto the narrator-cool bucket (Oduvan/Ro both came out as the narrator before this was fixed); the shared helper is `buildHintFromCast` in `server/src/tts/synthesise-chapter.ts`. `server/src/routes/voices.ts` reuses the same helper so the library aggregator and the synth path can never disagree on how the hint is projected.
- **The SSE handle lives in the store, not the view.** `generationStreamMiddleware` owns the cancel callback in a module-scope closure; opening / closing is driven by slice transitions (regen-spec lands, pause toggles, idle drains the queue, book switches), not React component lifetime. This is what makes generation continue running when the user navigates from Generate → Cast / Manuscript / Voices; the previous in-view `useEffect` cancelled the stream on unmount, which froze the mock state machine entirely (`mockStreamGeneration` is the worker — when nobody listens, no work happens) and the user came back to a stale-looking screen. Regen actions get an explicit pre-reconcile `closeHandle()` so the new spec is forwarded on the fresh stream instead of skipping over an already-open handle.
- **Heartbeat → stalled detection.** `chaptersState.lastTickAt` is `Date.now()` on every non-idle tick and `null` on idle (or before any tick). The Generate view and the global header pill both derive `stalled = (now - lastTickAt) > STALL_THRESHOLD_MS && hasInProgressChapter`. Threshold is exported from `chapters-slice.ts` (currently 30 000 ms) so it tunes in one place. Stalled flips the in-progress chapter row's "Generating" pill to a warning "Stalled" variant, surfaces a copy banner above the chapter list, and switches the header pill to amber. Without this the user has no signal between "worker is busy" and "worker has died on the floor."
- **Per-group "started" tick.** A single same-speaker group can be a multi-minute synth on CPU XTTS (one long narrator block folds into one TTS call). `synthesiseChapter` therefore fires `onGroupStart` **before** the `provider.synthesize` await, and the SSE route emits a `progress` tick from it carrying the first sentence of that group. Without this, only `onGroupComplete` ever fired, the SSE went silent for the duration of each call, and the 30s stall detector tripped for healthy in-flight work. Pinned by `server/src/tts/synthesise-chapter.test.ts` ("fires onGroupStart BEFORE the provider call for each group").
- **Activity feed on the Generate sidebar shows real events only.** The sidebar filters change-log events by `e.at` (ISO timestamp set at runtime by the middleware and the regen-confirm handlers) so the fixture seed in `src/data/change-log.ts` never leaks into the live view; the broader `#/log` / `#/books/:bookId/log` page still includes everything for historical browsing. The middleware emits `generation_started` on the first non-idle tick of a run (with the targeted chapter ids), `chapter_complete` when a `chapter_complete` tick lands, and `chapter_failed` when a per-chapter failure tick lands. `chapter_failed` is a new `ChangeLogType`; `LOG_TYPES` and `FILTER_MAP.generation` were updated to include it.
- **Global header pill** (`GenerationPill` in `top-bar.tsx`) is rendered by `Layout` whenever any chapter is `in_progress`/`queued` or `lastError != null`. Three variants: `running` (peach, spinner, "Generating · 3/12 · 42%"), `stalled` (amber, clock, "Stalled · 3/12"), `halted` (rose, warning, "Halted · 3/12"). The layout re-renders once per second while a run is alive (`setInterval`) so the stalled-derived state flips correctly even when the user is on Cast/Manuscript/Voices — without this re-render trigger, the slice-state-only memo would never invalidate. Clicking the pill routes back to `/books/:bookId/generate`.
- **Per-chapter failure row was promoted.** Previously a 1-line truncated rose caption under the chapter title; now a full-width rose card with icon, the verbatim `errorReason`, and an inline "Retry" button — same data, more legible. The truncated caption is gone so the same string isn't rendered twice.
- **Counters and the global pill count only non-excluded chapters.** The Generate view's "X of Y chapters complete" header, the "X of Y lines synthesised" sub-counter, and the top-bar `GenerationPill` `{done}/{total}` all derive from the active-only subset (`chapters.filter(c => !c.excluded)`). Excluded chapters never queue or synthesise, so leaving them in the denominator would lock the readouts at "N of N+excluded" forever and never reach the all-complete state — matching the behaviour `overallProgress` already enforces for the progress bar. Pinned by `src/views/generation.test.tsx` ("counters exclude ignored chapters") and `src/store/generation-stream-middleware.test.ts` ("counts only non-excluded chapters in the activeStream snapshot").
- **Drain check (`hasWork`) is excluded-aware too.** The same active-subset filter that scopes the counters also scopes `hasWork()` in `generation-stream-middleware.ts` — the per-tick reconcile pass that decides whether to close the SSE handle. Excluded chapters are reset to `state: 'queued'` by `setChapterExcluded` and reloaded as `'queued'` by `hydrateFromBookState` (they have no audio on disk), so without the `!c.excluded` filter a single pre-run exclude leaves a `'queued'` row that keeps the handle open forever. The user then sees the top-bar pill freeze at `Generating · N/N · 100%` after the last real chapter completes. Pinned by `src/store/generation-stream-middleware.test.ts` ("tears down the handle when the only remaining queued chapters are excluded").
- **Server `idle` tick is the cross-book end-of-stream signal.** `server/src/routes/generation.ts` emits exactly one terminal `{type:'idle'}` per run, then `res.end()`s every attached subscriber. The middleware's `applyGenerationTick` branch calls `closeHandle()` directly on idle — not via the reconcile drain check — because reconcile is gated on `currentBookId === handle.bookId` and returns early when the user is on a different book or a global view at the moment the idle tick arrives. Without this explicit close, an off-book idle left the handle (and the `activeStream` snapshot, and the global pill) live until the user navigated back. Pinned by `src/store/generation-stream-middleware.test.ts` ("closes the SSE on idle even when the slice has drifted to a different book").
- **"Include in book" on an excluded row runs subset re-analysis and triple-merges the response.** Clicking `+ Include in book` (the only action on an excluded-chapter row) flips the server-side `excluded` flag to false, optimistically updates the slice, and ALWAYS runs `POST /api/manuscripts/:id/analysis/chapters` for that chapter id — even when the chapter has cached attribution on disk. The trade-off is one extra Phase 0a+1 round-trip vs. the user needing to reason about cache freshness after manuscript / cast edits made while the chapter was excluded. On success the handler dispatches THREE slice updates: `castActions.mergeCharacters(res.characters)`, `chaptersActions.mergeSubsetAnalysis({ response, chapterIds:[id] })`, and `manuscriptActions.hydrateFromAnalysis(res)`. Pre-fix only the middle one was dispatched, so the chapter's freshly-attributed sentences never reached `manuscript.sentences` and audio generation had nothing to synthesise — the chapter looked included but was dead weight. Pinned by `src/views/generation.test.tsx` ("on success, merges sentences into the manuscript slice, characters into cast, and clears the row excluded flag").
- **Inline phase / progress / Cancel / error UI on the excluded-chapter row.** While subset analysis is streaming, the greyed-out row replaces the idle "+ Include in book" CTA with a phase label (`Re-analyzing — Detecting characters (Phase 0a)` → `…Parsing and attribution (Phase 1)`), a gradient progress bar driven by the SSE's `onPhase` ticks, an amber throttle banner when the analyzer hits an RPM / TPM / RPD wait, and a `Cancel` button that aborts the underlying `AbortController`. On failure the row morphs to a rose `Re-analysis failed: <message>` line with a `Retry` button. The transitional UI is keyed on a per-chapter `SubsetProgress` map kept in `GenerationView` component state, NOT the slice — the run only lives until the row transitions back to a normal queued row (success), reverts to the idle excluded variant (cancel), or stays on the error variant awaiting Retry. The `ChapterRow` excluded-or-transitioning predicate is `chapter.excluded || subsetProgress` — without the `|| subsetProgress` half, the row would morph into a normal queued row the instant `setChapterExcluded(false)` lands in the slice (which happens BEFORE the analysis call returns), losing the progress / Cancel / error UI mid-stream.
- **Rollback on cancel / error.** The slice and server flag are flipped optimistically (excluded → false) before the subset analysis runs, because the subset route hard-rejects requests for currently-excluded chapters (`server/src/routes/analysis.ts:1899`). If the call is aborted or fails, the chapter would otherwise drift to an included-but-unanalysed state that no UI flags. The handler therefore re-fires `setChapterExcluded(id, true)` and re-dispatches the slice action in the catch arm (best-effort; logged on rollback failure). After rollback, `chapter.excluded === true` and the row renders the errored variant (because `subsetProgress.error != null`) or the idle variant (for cancel). Pinned by `src/views/generation.test.tsx` ("Cancel aborts…", "on error, surfaces the message…").
- **Local-analyzer mid-generation guard.** The `+ Include in book` click is wrapped in `useLocalAnalyzerGuard`. When the user has picked a local Ollama model AND a generation stream is alive, the hook surfaces a `ConfirmDialog` ("Pause audio generation to analyse?") that dispatches `setPaused(true)` before invoking the un-exclude flow — the analyzer Ollama and the TTS sidecar compete for the same GPU, so kicking analysis mid-generation would silently evict TTS. Remote engines (Gemini, Gemma-via-Gemini) and idle TTS short-circuit straight to the call-through. Pinned by `src/views/generation.test.tsx` ("local-analyzer + active stream → confirm modal", "remote analyzer skips the modal"). Matches the sticky-generation invariant in [plan 31](31-sticky-generation.md).

## Acceptance walkthrough

Run with `VITE_USE_MOCKS=false`, sidecar (or Gemini) up.

1. **Start generation from a freshly confirmed cast** → URL becomes `#/books/<id>/generate`; SSE stream opens; chapters list each show queued state for their characters.
2. **Per-character progress** — the active speaker's slot shows `in_progress` driven by the tick's `characterId` (no client-side threshold inference); when the next group's speaker is different, the previous active slot is set back to `queued`. The slot's bar width is always the truthful "lines of this character done so far" fraction derived from the manuscript, so characters who have spoken some but not all of their lines show partial green progress instead of either "Done" or "Queued". `chapter_complete` then flips every non-skipped character to `done` and pins the bars full. `currentLine / totalLines` updates inside `in_progress`.
3. **Assembling phase** — after the last per-group `progress` tick, a `chapter_assembling` tick fires. The chapter row shows a neutral ink-toned striped bar instead of the magenta synthesis gradient and an "Assembling…" pill; the expanded row shows "Writing chapter file…". Resolves to `done` on `chapter_complete`.
4. **Chapter complete** — chapter card shows "complete" badge; the MP3 exists on disk under `books/<Author>/<Series>/<Title>/audio/<chapter-slug>.mp3`. The next queued chapter starts automatically the moment the server emits its own `progress` tick for it; nothing on the client gates the promotion.
5. **Preview playback** — click Preview on a done row → `setCurrentTrack(chapterId)` dispatched; MiniPlayer mounts a real `<audio>` element backed by `GET /api/books/:bookId/chapters/:chapterId/audio.mp3` (range-supported, so seek works) and renders the segment colours from `GET .../audio` (JSON ChapterAudio).
6. **Per-chapter failure** — kill the TTS provider mid-chapter. Stream emits `chapter_failed` with `chapterId + errorReason`; that row goes red with the reason inline. Subsequent queued chapters are unaffected — the server may move on to the next chapter, in which case its `progress` ticks promote it as normal.
7. **Stream-level failure** — point the engine at a bad `modelKey` (or down the sidecar). Stream emits `chapter_failed` **without `chapterId`**; the dismissible error banner appears above the chapter list; the in-flight chapter flips to `failed`; queued chapters re-pill as "Blocked" until the banner is dismissed.
8. **Cancel mid-generation** — click Cancel / navigate away → `controller.abort()` fires; no error toast surfaces (per the abort exception filter). The next chapter does not start; in-progress chapter may finish if the server has already kicked off synthesis.
9. **Regenerate (this/forward/character/batch)** — one of the three regenerate reducers populates `pendingRegen` + bumps `regenEpoch` + clears `lastError` and `generationStartedAt`. The Generate view's `useEffect` re-opens the SSE with `{ chapterIds: pendingRegen.chapterIds, force: true }` **and immediately drains `pendingRegen` via `consumePendingRegen`**. `regenEpoch` keeps incrementing so a follow-up regenerate of the same target re-fires. Pause→Resume after regenerate must NOT re-force-regen the original target set: the drain on open is what makes the Resume reopen a "normal" resume (no `chapterIds`, no `force`) so it picks up the next chapter that lacks audio on disk.
10. **Subset run** — start with `chapterIds: [5, 7, 9]` → only those chapters enter `in_progress`; others stay queued/unaffected.
11. **Idle tick** — when no chapter is in progress (mock mode polling or server breather), an `idle` tick fires; UI does not interpret it as failure. `pendingRegen` is cleared; `generationStartedAt` is cleared only if no chapter is `in_progress`/`queued`.
12. **Chapter & character metadata visible from the queued state on.** Each chapter card shows `X words · Y lines · Z speakers` under its title before generation has emitted any tick. Expand any chapter row and each character entry shows `<lines> · <words>` next to its name. These figures are stable across pause/resume/regenerate because they come from `state.manuscript.sentences`, not the SSE.
13. **Overall progress includes hydrated-Done chapters.** Re-open a partially generated book where some chapters already have `.mp3` on disk: the Generate view shows them as Done with no live ticks ever fired, and the Overall-progress bar reflects their full share of the manuscript (sentence-weighted). Regression for the "3/7 done shows 4 %" bug — see `src/views/generation.test.tsx`.
14. **Background generation across view nav.** Start a generation run, then click Cast / Manuscript / Voices / Activity mid-flight. The peach "Generating · N/T · X%" pill stays in the top-right of every view, percent tick visibly increments, and clicking it returns to Generate where chapter progress matches the pill. ## Out of scope

- Chunked streaming for live playback during generation — current path encodes the whole MP3 then signals complete (atomic write via `<slug>.mp3.tmp-…` → `rename`).
- Multi-book parallel generation — single-book scope per stream.
- Resume after server restart — the next POST decides what's missing on disk and resumes from there.
