# Profile drawer

> Status: stable
> Key files: `src/modals/profile-drawer.tsx` (incl. `ModelVoiceOverridePicker`), `src/store/cast-slice.ts` (`updateCharacter`, `renameCharacter`), `src/store/voices-slice.ts` (`setOverride`, `hydrateBaseVoices`), `src/lib/tts-voice-mapping.ts` (`resolveTtsVoiceForCharacter`), `src/lib/api.ts` (`setVoiceOverride`), `src/store/ui-slice.ts` (`setOpenProfileId`), `src/store/persistence-middleware.ts` (`cast/renameCharacter` persist rule), `server/src/routes/voices.ts` (`applyOverrideToCastFiles`)
> URL surface: `?profile=<characterId>` inside `#/books/:bookId/:view` and `#/books/:bookId/confirm`
> OpenAPI ops: `POST /api/voices/:voiceId/sample`, `PUT /api/voices/:voiceId/override`

## What this covers

Edit drawer for a single character. Lets the user adjust identity (name, role, gender, age range), tone (warmth, pace, authority, emotion sliders), description text, and evidence quotes. Triggers live recomputation of the mapped TTS voice and a per-character sample preview. The drawer is URL-addressable via `?profile=` so a deep link reopens it. Reachable from both the ready-stage Cast view and the confirm-stage "Meet the cast" cards so identity / gender / age can be corrected before generation starts.

## Invariants to preserve

- Opening the drawer dispatches `setOpenProfileId(characterId)`; the URL gains `?profile=<id>` (per `01-hash-router.md`).
- `setOpenProfileId` is accepted on the `ready` and `confirm` stages (where `openProfileId` lives on the stage variant). Dispatching from any other stage is a no-op.
- Tone fields are integers in `[0, 100]`. Gender enum: `'male' | 'female' | 'neutral'`. Age range: `'child' | 'teen' | 'adult' | 'elderly'`.
- Evidence list shows the first **3** quotes by default; "Show more evidence" toggle reveals any quotes beyond 3. Each quote: `{ quote: string; note?: string }`. The array is **sorted longest-first server-side** by `sortEvidence` in `server/src/routes/analysis.ts` so the first quote is both the prominent UI excerpt and the voice-cloning sample source. The drawer does NOT re-sort — it trusts the persisted order.
- Evidence drives the voice sample: `server/src/routes/voice-sample.ts` `buildSampleText` consumes the longest quote (typically index 0) for the ~12-second cloning preview. The analyzer skill (`skills/audiobook-character-analysis.md`) asks for ≥3 quotes per character with one ~180–280 char excerpt, so the longest is intentionally substantial. Short rosters fall back to a `<Name> said: …` pad; empty evidence falls back to the generic "Hello. I'm …" script.
- `resolveTtsVoiceForCharacter` recomputes the mapped TTS voice live as the user edits gender/age. The mapping logic lives only in `src/lib/tts-voice-mapping.ts` and its server twin in `server/src/routes/voice-sample.ts` — keep them in sync.
- Preview / sample requests POST `/api/voices/<voiceId>/sample` with `{ modelKey, voice, text?, characterHint }` (`src/lib/api.ts:92-108, 518-530`). `characterHint` includes `description`, `role`, `gender`, `ageRange`, `evidence[]`, `tone`.
- `VoiceSample` response shape: `{ url, durationSec, cached, modelKey }`. UI plays the URL via `<audio>` and shows the cached badge if `cached: true`.
- **JIT TTS auto-load on Play** (`src/lib/play-sample-with-auto-load.ts`) — the Play button funnels through `playSampleWithAutoLoad`, NOT a direct `api.getVoiceSample` call. The helper probes `/api/sidecar/health`, evicts the analyzer (via `api.unloadAnalyzer` — and only when `/api/ollama/health` reports `modelResident: true`), loads the sidecar (`api.loadSidecar`), then synthesizes. Mirrors `generation.tsx` `handleLoadTts` so the "model lifecycle is button-driven, not eager" rule (CLAUDE.md) is preserved — the user's Play click is the load consent. Status sequence emitted via `onStatus`: `evicting` → `loading-tts` → `synthesizing`. Concurrent clicks share a single in-flight prep promise so a Drawer + Cast row click doesn't double-evict. An inline eviction banner ("Analyzer unloaded to free VRAM for TTS.") surfaces only when the analyzer was actually resident. The same helper backs `src/views/cast.tsx` per-row Play. Tested by `src/lib/play-sample-with-auto-load.test.ts`.
- **VoiceSwatch is a play affordance everywhere it renders** (`src/components/primitives.tsx`) — the gradient circle has a hover-revealed play icon; clicking it MUST route through `playSampleWithAutoLoad` at every call site (cast row, profile drawer, match-detail modal). The primitive exposes `onSelect` + `loading` props; `loading` swaps the play icon for `IconSpinner`, sets `aria-busy`, and disables the button so re-clicks can't fire a concurrent request. Accessible name is `Play sample for <character>` (or `Generating sample for <character>` while busy). In the **cast row** the swatch click intentionally bubbles to the row's `onClick` so a single click opens the profile drawer AND starts playback in one gesture; the drawer's own swatch coalesces with the cast row's via the in-flight gate in `play-sample-with-auto-load.ts`. Tested by `src/views/cast.test.tsx` and `src/modals/profile-drawer.test.tsx`.
- The drawer never directly mutates `cast.characters`; it dispatches `castActions.updateCharacter({ id, patch })` so the change is funneled through one reducer (Immer-friendly).
- **Model-voice override picker** (`ModelVoiceOverridePicker` in `src/modals/profile-drawer.tsx`) — sits beneath the Preview voice block and lets the user pin a specific base voice for this character, bypassing the attribute-driven match.
  - **Engine-tabbed**: one tab per engine present in `state.voices.baseVoices` (Coqui, Kokoro, optionally Gemini). The active tab determines which engine's catalog the picker exposes; switching tabs preserves the OTHER engines' selections. Tabs render a small `.bg-magenta` dot when that engine has a non-null slot, and the project's currently-active engine carries a small "Active" label so the user knows which slot actually drives synthesis right now.
  - **Searchable picker** (`src/components/voice-override-picker.tsx`) — wraps the shared `<SearchablePicker>` primitive (`src/components/searchable-picker.tsx`). Trigger button shows the resolved Auto label or the picked voice name + chevron; clicking opens a portal popover with a "Search voice…" autocomplete input + filtered list (Auto row first, then voices for the active engine tab). Replaces the previous native `<select>` which scrolled past 40+ Kokoro voices unfiltered.
  - The selected value round-trips through string values: `AUTO` or `${engine}|${name}`. The selected value for the active tab is read from `voice.overrideTtsVoices?.[engineTab]`; legacy `voice.overrideTtsVoice` is folded in by `mapOverridesToBaseVoiceMap` so older Voice payloads still render correctly during the migration window. When the active tab matches the auto-resolved engine the Auto row reads "Auto — currently &lt;Engine&gt; · &lt;Name&gt;"; for other tabs it reads "Auto for &lt;Engine&gt; — attribute-driven".
  - The base-voice catalog is sourced from `state.voices.baseVoices`, hydrated by `api.getBaseVoices()` on Voices-view mount (see plan 22). The drawer is read-only on that hydrate — it does not trigger the fetch itself; until the catalog loads the picker is disabled and shows "Loading base voice catalog…".
  - `onChange` is optimistic and uses **deferred rollback**: dispatch `voicesActions.setOverride({ voiceId, override })` first (slice mutation no-ops when no library Voice exists yet for this character; the reducer writes into `overrideTtsVoices[override.engine]` while preserving other engine slots), then `await api.setVoiceOverride(voiceId, override)`; on failure, restore the previous slot map by re-dispatching every engine's saved override.
  - **Per-engine semantics replaces the old engine-mismatch warning** — because each engine has its own slot, switching the project engine no longer "loses" the manual cast assignment; the right slot just lights up. The mismatch banner is gone.
  - **Server propagation** — `PUT /api/voices/:voiceId/override` with `{ override: { engine, name } }` walks every confirmed-cast `cast.json` and writes the slot into `overrideTtsVoices[engine]`, preserving any existing slots for other engines (`server/src/routes/voices.ts`, `applyOverrideToCastFiles`). `{ override: null }` clears ALL engine slots. The same character recurring across a series stays in sync; one PUT updates every cast.json atomically and 404s if no character matches. Legacy singular `overrideTtsVoice` is folded into the new map at read time and removed on the next write — one user action upgrades the on-disk shape.
- **Merge / downgrade controls** — the Cast roster section renders two cohabiting affordances when `onMerge` is wired:
  - A picker (`Merge <name> into another character…`) for folding one identity into another real character. Hidden when `mergeCandidates` is empty.
  - **Merge target is a searchable picker** — the `Keep which character as the survivor?` trigger button opens a `<CharacterSearchPicker>` popover (the same shared component the manuscript-view's reassign picker uses). The picker autocompletes against both in-book siblings and the prior-books series roster; prior-roster picks route through the picker's `onPickRosterEntry` opt-in so the merge handler can fire `onLinkPrior` (manual continuity link) instead of the materialise-then-assign flow the manuscript view uses. Selecting an in-book sibling still routes through `onMerge`. Pre-extraction this was a native `<select>` with `<optgroup>` and no typeahead.
  - Two direct buttons ("Unknown male" / "Unknown female") for downgrading the current character into a standing background bucket. Available even when the cast has no other candidates because `POST /api/books/:bookId/cast/merge` synthesises the bucket on the fly using `makeBucket()` from `server/src/analyzer/fold-minor-cast.ts` when the target id (`unknown-male` / `unknown-female`) isn't on `cast.json` yet.
  - Downgrade buttons are hidden for the buckets themselves (`unknown-male`/`unknown-female`) and for the narrator (`narrator`). Picker and buttons share `mergeBusy` / `mergeError` so two folds can't fire concurrently. On success the layout dispatches `castActions.applyMerge` with the server's recomputed roster and closes the drawer.
- **Rename + promote-alias** — the header name is editable and every "Also known as" chip can be promoted to the primary name. Both flow through one reducer, `castActions.renameCharacter({ characterId, name })`, which is **dispatch-only** (no API call): the `cast/renameCharacter` persist rule (`src/store/persistence-middleware.ts`) PUTs the cast slice to `cast.json` — the same file the merge/alias endpoints own — so the rename round-trips with no new server endpoint.
  - **Always lossless** — `renameCharacter` unconditionally demotes the OLD primary name into `aliases` (a rename never loses a name; earlier chapters may still use the old title). If the new name is itself an existing alias (the promote case), it's stripped from the aliases list so it doesn't double up. Dedup is case-insensitive + trim-tolerant; display casing is preserved. No-op on empty/whitespace, unknown id, or a name unchanged apart from case/whitespace.
  - **Header rename** (free-text): the name `<h3>` carries a `Rename` affordance (gated `onRename && !isBucket`). Clicking reveals an inline input (`aria-label="Character name"`) seeded with the current name; Enter or the `Save name` button commits, Escape cancels, empty input shows "Name cannot be empty." and does not dispatch. Mirrors the `+ Add alias` focus/keydown pattern.
  - **Promote an alias** (`Make <alias> the primary name` star on each chip): one tap swaps that alias up to the primary name and demotes the previous name to a chip — the recovery path for a coarse/wrong merge that stranded the fuller name as an alias.
  - **Buckets vs narrator**: rename + promote are hidden for the `unknown-male`/`unknown-female` buckets (auto-synthesised — a rename is meaningless). The **narrator IS renameable** (a book may legitimately name its narrator; attribution is by id, so it's safe). `Character.name` is never a lookup key — sentences attribute by `characterId` — so a rename can't break attribution.
  - **Activity-log entry** (`fe-16`): the layout's `onRename` wiring also dispatches a `name_change` change-log event (`buildNameChangeEvent`, title `Renamed <old>`, note `Now known as "<new>"`) — mirroring `voice_tune`/`voice_lock`. It's categorised under **Cast** in both the per-book Activity view (`FILTER_MAP` in `src/views/change-log.tsx`) and the workspace aggregator (`CATEGORY_MAP` in `server/src/routes/workspace.ts`), rendered with `IconPencil` via `LOG_TYPES`. The same no-op guard the reducer uses gates the dispatch, so a case-only rename emits nothing. Persisted by the existing `changeLog/appendLogEvent` rule.

## Acceptance walkthrough

Run `VITE_USE_MOCKS=true`, navigate to a book's cast view.

1. **Click a character card** → drawer slides in; URL becomes `#/books/<id>/cast?profile=<characterId>`.
2. **Reload the page** → drawer reopens automatically from the URL on hydration.
3. **Rename a character** → click the `Rename` affordance next to the name in the header, type a new name (e.g. "Dame Linnet" → "Councilor Linnet"), press Enter. The header + cast card title update immediately; the old name ("Dame Linnet") appears as an "Also known as" chip. Reload → the new name persists (round-tripped to `cast.json`). An empty name is rejected with "Name cannot be empty."
3a. **Promote an alias to the primary name** → on a character with aliases (e.g. after a merge folded "Dexter Alvin Diznee" into "Hart"), click the star (`Make <alias> the primary name`) on the fuller alias chip. The header becomes that name and the previous primary ("Hart") demotes to a chip. Reload persists. The `Rename` affordance and the star are hidden for the `unknown-male` / `unknown-female` buckets.
4. **Drag warmth slider** to 80 → `tone.warmth = 80` in slice; the sample-preview button stays enabled.
5. **Verify evidence order**: the first 3 quotes are visible by default, longest first. When the character has ≥4 quotes, **toggle "Show more evidence"** to reveal the rest; toggling off collapses back to the first 3. State persists for the session (not URL-encoded).
   - Spot-check: the topmost quote is the longest in the array (server enforces this via `sortEvidence`).
6. **Click "Preview voice"** → POST `/api/voices/<voiceId>/sample` fires with the current `characterHint`. Response plays inline; second click within minutes returns `cached: true` and shows a "cached" badge.
7. **Switch TTS model in the engine picker** (see `13-tts-engine-picker.md`) → next preview includes the new `modelKey` and bypasses cache for that key.
8. **Edit gender from `'male'` to `'female'`** → `resolveTtsVoiceForCharacter` recomputes; the displayed mapped voice name updates. Preview fetches a new sample for the new voice.
9. **Override the model voice via the engine-tabbed picker**:
   - Open the drawer. Engine tabs render across the top of the picker (Coqui, Kokoro, optionally Gemini); the project's active engine carries an "Active" label. The `<select>` below the tabs shows "Auto — currently &lt;Engine · Name&gt;" for the active engine's tab.
   - On the **Coqui** tab, pick a specific speaker (e.g. Asya Anara). Network: `PUT /api/voices/<voiceId>/override` with `{ override: { engine: 'coqui', name: 'Asya Anara' } }`; response 204. The Coqui tab now renders the filled-slot dot. The mapped-voice line above and the cast card's swatch update immediately (optimistic).
   - Switch to the **Kokoro** tab — the picker resets to AUTO for Kokoro (the Coqui slot is preserved, just not displayed here). Pick a Kokoro voice (e.g. `am_onyx`). Now both tabs show the filled-slot dot. Reload the page → both overrides persist.
   - **Cross-book propagation**: open a different book that contains a character with the same `voiceId` (e.g. a series recurrence) → that character's profile drawer shows the same Coqui AND Kokoro overrides pre-selected. Server's `applyOverrideToCastFiles` updated every cast.json's `overrideTtsVoices` map.
   - **Engine switch preserves cast** (the killer feature): change the project's TTS engine via the engine picker (plan 13). The override doesn't go missing — the cast view's voice resolution just picks up the other tab's slot. No re-cast needed when toggling Coqui ↔ Kokoro.
   - **Legacy migration**: if a cast.json was written by an older client carrying the singular `overrideTtsVoice` field, the drawer still displays it correctly (read-time `mapOverridesToBaseVoiceMap` projects it into the engine map). The next override write upgrades the on-disk shape and drops the legacy field.
   - **Clear-to-Auto is currently all-engines**: pick "Auto" on any tab and the network call is `{ override: null }`, which clears EVERY engine slot — the other tab's filled-slot dot disappears too. Per-slot clearing isn't surfaced in the API yet (would need `{ engine, name: '' }` to be accepted as a delete-this-slot signal); track it as a known gap. The picker returns to "Auto — currently &lt;Engine · Name&gt;" on every tab; server-side the entire `overrideTtsVoices` field is removed from each cast.json entry.
10. **Downgrade a descriptor-named speaker** (real backend only — mock returns an empty cast):

- From the "Meet the cast" confirmation view, click a card like _Rescuer_ (≥3 lines so the auto-fold left it alone).
- In the drawer, scroll to **Cast roster** → click **Unknown male** or **Unknown female**.
- Network: `POST /api/books/:bookId/cast/merge` with `{ sourceId, targetId: 'unknown-male' | 'unknown-female' }`. Response is the recomputed roster; the source disappears and the bucket gains the source's name in its `aliases` list.
- Drawer closes; URL drops `?profile=`. The roster card for the bucket now reflects the merged line/scene count.
- Re-open the bucket's profile → the downgrade buttons are HIDDEN for it (a bucket can't be downgraded into itself).

11. **Close drawer** → `setOpenProfileId(null)`; URL drops `?profile=`.

## Out of scope

- Voice library browsing while the drawer is open (separate view).
- Cross-character bulk edits — covered by `11-batch-character-regenerate.md`.
- Audio waveform of the preview — `<audio>` element only in v1.
