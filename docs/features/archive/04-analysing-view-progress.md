# Analysing view & SSE progress

> Status: stable
> Key files: `src/views/analysing.tsx`, `src/lib/api.ts` (`realAnalyseManuscript`, `AnalysisError`)
> URL surface: `#/books/:bookId/analysing`
> OpenAPI ops: `POST /api/manuscripts/:id/analysis` (SSE stream)
> Related: [32-sticky-analysis.md](32-sticky-analysis.md) for the server-owned job lifecycle + the `AnalysisPill` cross-navigation surface (B-series).

## What this covers

Server-sent-events stream rendering for the two-stage analysis pipeline. Phase ticks drive a multi-step progress bar; structured `live` payloads on each phase tick surface the active chapter + elapsed-of-estimate ETA so the user sees liveness between log entries. The view supports model selection (via the re-parse dialog) and a "Start fresh" toggle that discards any cached partial progress.

## Invariants to preserve

- **Overall-progress math is centralised** in `src/lib/analysis-progress.ts`
  (`computeOverallProgress(phaseId, phaseProgress)`). The header
  `AnalysisPill` (`src/components/layout.tsx`) AND the analysing view's
  "Overall" bar (`src/views/analysing.tsx`) read from this one helper.
  Phase weights are `[0.45, 0.5, 0.05]` — Phase 0 and Phase 1 carry
  most of the cost, Phase 2 is a quick library reconciliation. A
  module-load guard catches future drift between `PHASE_WEIGHTS` and
  `ANALYSIS_PHASES`. Regression motivator: the bar previously used a
  naive `(phase + phaseProgress) / 3` average that disagreed with the
  pill by 15 percentage points (55% vs 40%) on the same stream state.
- Stream event union: `{ kind: 'phase' | 'log' | 'result' | 'error' | 'chapter-failed' | 'chapter-resolved' | 'cast-update' | 'heartbeat' | 'eta', ... }` (`src/lib/api.ts`). Frontend handlers must accept each kind; unknown kinds are ignored, not thrown. `chapter-resolved` fires whenever the server clears a chapter id from `cache.failedChapterIds` — emitted by both the full route's Phase 0a re-queue success path and the subset retry route. The view drops the matching Retry row in response so the panel never lags behind the cache.
- `live` payload shape: `{ totalChapters, chapters: AnalysisLiveChapter[] }` where each in-flight chapter is `{ chapterIndex, chapterTitle, elapsedMs, estMs }` (`src/lib/api.ts:55-70`). The server emits one entry per chapter currently in flight, sorted by `chapterIndex` ascending; the UI renders one row per chapter so a stuck chapter doesn't visually mask the others making progress.
- Request body is omitted entirely when neither `model` nor `fresh` is set; included only when one is (`src/lib/api.ts:424-429`). Do not always send a body — manual-mode servers may not parse JSON.
- `AnalysisError` carries `code` (e.g. `'rate_limit'`, `'auth'`, `'unknown'`) and optional structured `detail` (e.g. Google's `status` + `details[]`) (`src/lib/api.ts:412-421`). UI surfaces the headline message inline and the `detail` in a collapsible block.
- SSE frame format is `data: <json>\n\n`; multiple `data:` lines per frame join with `\n` (`src/lib/api.ts:458-468`). Do not change the splitter.
- "Start fresh" sets `fresh: true` in the request body; the server discards any partial progress before re-running (no resume).
- **Phase-card detail copy spans the full card width, not the column beneath the label.** In `src/components/analysing/phase-card.tsx` the per-phase model chip + swap dropdown (plan 95) share the label's `justify-between` flex row, but the `detail` paragraph lives *outside* that row so it never gets squeezed into the narrow column to the left of the controls (where it wrapped into a cramped two- or three-line block). Locked by `src/components/analysing/phase-card.test.tsx` — the detail must not be a descendant of the chip's flex row. Regression motivator: the "Detecting characters" / "Parsing and attribution" details rendered as a tight ragged column under the label while the right half of the card sat empty below the dropdown.
- **Phase 0 cast roster persists as a visible outcome.** Once `cast-update` events have populated the cast slice, the chip list under the Phase 0 row stays visible even after Phase 0 completes — including the cached-resume fast path that fires after a model switch, where the server only re-emits a single "Phase 0 already complete" log line. The roster is Phase 0's outcome and must not vanish when the active phase advances.
- **Phase 1 cannot start while any chapter is missing its cast.** When `cache.failedChapterIds` is non-empty at the end of Phase 0a, the main analysis route emits `{ kind: 'error', code: 'cast_incomplete', ... }` and ends without writing `cache.stage1` (`server/src/routes/analysis.ts` — search "Phase 1+ MUST NOT advance"). The view treats this code specially: no red error banner, `castIncomplete` flag set, retry buttons in the failed-chapter panel are clickable, and once `failedChapters` drains to 0 the auto-resume effect re-fires `/analysis/stream` so Phase 1 picks up automatically.
- **Stage 1 is finalised only when every non-excluded chapter has a non-empty per-chapter roster on disk.** The subset-retry path (`POST /api/manuscripts/:id/analysis/chapters`) used to gate stage1 writes on `failedChapterIds.length === 0` alone, but that predicate is wrong when `chapterCast` covers only a fraction of the book — a sparse cache plus an empty failed list would let `rebuildRoster()` overwrite a richer existing stage1 with a partial roster. The route now also checks coverage via `isPhase0aCoverageComplete(chapterCast, chapterHints)` (`server/src/routes/analysis.ts`); when coverage is incomplete it emits `{ kind: 'error', code: 'cast_incomplete', ... }`, leaves any existing stage1 intact, and skips Phase 1. The view's `cast_incomplete` handling already drops the user into the "Paused — run main analysis to fill gaps" flow. Regression motivator: "Unlocked" (mns_VoP0mLGvov) — chapterCast for 28/182 chapters, failedChapterIds=[], a subset retry rebuilt stage1 from Narrator-only entries and clobbered a 6-character roster.
- **Every `manuscript-edits.json` `characterId` is a member of `stage1.characters`.** Phase 1 sentence attribution is supposed to honour this constraint (per `audiobook-sentence-attribution.md`'s skill prompt), but the model occasionally fabricates ids and a later stage1-shrink can orphan ids that _were_ valid at attribution time. Both Phase 1 end-of-route write sites (main route `analysis.ts:~1860`, subset route `analysis.ts:~2310`) now run `reconcileSentenceCharacterIds(folded.sentences, validIds)` before persisting; unknown ids are demoted to `narrator` and a per-id summary lands in the analysing-view log. If the demotion rate exceeds 5% on a sample of ≥100 sentences, the route emits `{ kind: 'error', code: 'attribution_drift', ... }` instead of the usual `result` event, skips the cast.json / state.json writes (so the book stays in `analysing`, not `cast_pending`), and the user retries. `manuscript-edits.json` is still written with the demoted sentences so disk state stays internally consistent. The Unlocked-shaped regression (153 demoted of 4192 ≈ 3.65%) stays under threshold and advances to confirm — a single-voice audiobook is a degraded-but-coherent outcome that beats hard-stopping.
- **Stage 1 cannot silently shrink by >50% when a non-trivial roster (≥3) is already cached.** Every `cache.stage1 = stage1` write site in `server/src/routes/analysis.ts` passes through `stage1ShrinkRefused(prevCount, nextCount)` first. When the gate triggers and the request did not include `allowStage1Shrink: true`, the route emits `{ kind: 'error', code: 'stage1_shrink_refused', prevCharCount, nextCharCount }`, leaves the existing stage1 intact on disk, and ends the stream. The analysing view catches this code specifically — no red error banner, instead a yellow `data-testid="stage1-shrink-refused-banner"` card with the prev/next counts and an "Accept smaller roster" button that re-fires the request with `allowStage1Shrink: true` (bypasses the gate for that attempt). Regression motivator: Unlocked silently went from 6 detected characters to 1 when a subset retry with Gemini 3.1 Flash Lite landed on chapters the model collapsed to Narrator-only — the user discovered the loss only when "Cast so far · 1 character" sat under Detecting characters with no warning.
- **Detection treats journals / medical logs / registry files / bios as first-person speech by their author, not narrator prose.** The skill at `skills/audiobook-character-detection-per-chapter.md` and the per-chapter inbox template both call this out explicitly: when a chapter is a journal entry, medical log, registry file, diary, letter, transcript, or bio page AND the author is named or strongly implied (chapter title, header `FILED BY: X`, signature `—X`, surrounding bio block), the _author_ becomes the character on the roster with their `id` set to their name and the document's prose as their evidence. `id: "narrator"` is reserved for omniscient third-person prose with no in-fiction author. Regression motivator: Unlocked's per-chapter detector collapsed all 28 cached chapters to Narrator-only (Oduvan's medical logs, Wren's Registry bio, Marlow's journal entries — each treated as a different "Narrator" with a one-off description) because the original dialogue-only inclusion test didn't recognise document authorship as speech.
- **Series-cast prior carries identity across sibling books, deduped per unique character.** When analysis kicks off on a book that belongs to a series, the route resolves `scanSeriesCharactersForBookId(record.bookId)` (`server/src/workspace/series-cast-scan.ts`) once, then folds the per-book rows through `dedupSeriesPrior()` (`server/src/workspace/series-prior-dedup.ts`) so a series-regular who appears in N prior books contributes one entry, not N. The dedup rule mirrors the prompt's own matching contract (name OR alias, case-insensitive, ignoring punctuation) and unions transitively via union-find so alias chains (Book A "Wren" ↔ Book B alias "Wren Sparrow" ↔ Book C name "Foster") collapse into a single group. The per-chapter prompt renders a `## Known characters from prior books in this series` section listing each unique character's `id` / `name` / `aliases` / `fromBookTitles` (an array of every source-book title); the skill paragraph tells the model to reuse those ids verbatim when a chapter speaker matches by name or alias. Standalones / first-in-series books resolve to an empty list — the prompt simply omits the section. Server also emits a one-shot `{ kind: 'series-prior', count, names: [first 3] }` SSE event at Phase 0 entry — the `count` is the deduped (unique-character) count, not the raw per-book row count; `src/views/analysing.tsx`'s `SeriesPriorPill` (data-testid `series-prior-pill`) renders that count and the first three names under the "Detecting characters" phase card. The pill consumes `s.analysis.activeStream.seriesPrior` from the analysis slice so it survives reload + cross-navigation; subsequent POSTs that attach to the same job receive the cached event via the catch-up replay (the server tracks `series-prior` in `replay.lastSeriesPrior` — see plan 32). Subset-retry (`POST /:id/analysis/chapters`) resolves and dedupes the same prior so a single-chapter re-analysis still recognises series-regulars. The Profile Drawer's manual continuity-link picker (`GET /api/books/:bookId/series-roster`) intentionally hits the un-deduped scan since it needs per-book provenance for the "fold this duplicate into Wren-from-Keeper specifically" flow.
- **Failed-chapter Retry pauses the main run for the subset call.** Both routes load their own snapshot of the disk-backed `analysis-cache.json` and write back independently, so running the two SSEs in parallel raced cache writes — the second finisher's stale snapshot of `cache.failedChapterIds` / `cache.chapters` clobbered the first's progress, and on reload the user saw their just-retried chapter unparsed and the failed row resurrected. Today the click aborts the in-flight main run's `AbortController` before firing `runAnalysisForChapters`, then re-arms the main run (sets `analysisStarted=true` and bumps `retry.nonce`) once the subset settles. The cast-incomplete auto-resume path is left untouched: when the main run hit `cast_incomplete` the controller ref is a zombie and `conn === 'idle'`, so `pausedMainForRetry` evaluates false and that flow keeps using the existing `failedChapters.length === 0` gate. Only one chapter can retry at a time (`retryingChapterId` tracks the active one).
- **Dropped-quote ledger is append-only.** Each Phase 0 verify pass that drops at least one quote appends a batch to `.audiobook/dropped-quotes.json` (envelope shape in `server/src/store/dropped-quotes.ts`). Both analysis routes write — they share the `persistDroppedQuotesBatch` helper so the field names stay aligned. Quotes are capped at `MAX_QUOTE_CHARS` (2000) with a `truncated: true` flag; reasons are enumerated (`not_in_source` | `empty_after_normalisation`) from the single conditional in `verifyEvidenceAgainstSource`. The analysing view's `DroppedQuotesPanel` reads the latest batch only via `GET /api/books/:bookId/dropped-quotes` and groups entries by character — no Restore button in Phase 1; this is a read-only audit surface for tuning the verifier prompt.
- **Verifier accepts quotes via three tiers** (`server/src/util/text-match.ts` → `matchQuoteInSource`). Strict substring is tier 1; tier 2 strips trailing `.,;:!?` from the candidate (handles "model writes `.`, source writes `,` before a dialogue tag" — the dominant Gemini / qwen3.5 false-positive); tier 3 splits the candidate on sentence-final punctuation and requires every segment (≥ 8 chars, ≥ 2 segments overall) to appear in source (handles "stitched" same-speaker utterances with the narration tag removed between halves). When tier 2 or 3 carries a quote, the verifier emits a single aggregate log line "Quote-match tiers: verbatim=N, terminal-punct=N, segments=N." so the analysing-view log reveals how often the looser tiers fire — feedback for tightening the thresholds without spelunking through cast.json.
- **Descriptor-name preview-fold + evidenceless drop at Phase 0b** (`server/src/analyzer/fold-minor-cast.ts` → `isDescriptorName`, `foldMinorCast({ nameOnly: true })`; `server/src/routes/analysis.ts` → `previewFoldForLiveView`, `dropEvidencelessCast`). Three patterns are treated as descriptor-as-name (not proper names): `^Unknown\b…` (the Stage-1 contract), `^The <Word>($| <Word>$)` (e.g. "The Jogger", "The Old Man"), and `<word> (Boy|Girl|Man|Woman|Guy|Lady|Kid|Person|Figure|Stranger|Voice)$` (e.g. "Drooly Boy", "Ponytail Girl"). All collapse into the `unknown-male` / `unknown-female` buckets the moment they appear in a chapter's roster — the live SSE `cast-update` payload, the on-disk `.audiobook/cast.json` mid-run write, and the Phase 0b `cast-update` all run the same name-only fold so the user sees one consistent roster across SSE and disk. The line-count fold and zero-line drop stay off in name-only mode (no stage-2 data yet); the full post-Phase-1 fold still runs against `stage1.characters` for authoritative counts. Phase 0b additionally drops non-narrator characters left with zero evidence after the verifier ran — those failed the skill's own inclusion test ("Test for inclusion: can you copy a verbatim sentence … that is dialogue the entity speaks?") and would otherwise persist as hollow cast entries the post-fold line-count rule can't catch when the model also fabricated stage-2 attributions for them. Narrator is exempt (its "lines" are prose, not dialogue).

## Acceptance walkthrough

Run with `VITE_USE_MOCKS=false` (server on `:8080`, `ANALYZER=local` or `ANALYZER=gemini`).

1. **Land on `#/books/:bookId/analysing` after upload** → stream opens; phase 1 progress bar starts ticking. Log lines appear under the active phase (e.g. "Detected 23 characters across 14 chapters").
2. **Live ETA** — during stage-2 chapter processing, the `live` block shows one row per in-flight chapter, e.g. `"Chapter 2/7 · DAY ONE · 4:15 of ~0:21 over budget"` on top of `"Chapter 7/7 · DAY SIX · 0:08 of ~2:03"`. Rows are sorted by chapter order so a slow chapter cannot hide concurrent progress.
3. **Stream ends with a `result` event** → frontend dispatches `analysisComplete({ bookId })`; URL transitions to `#/books/:bookId/confirm`.
4. **Force a server error** (kill the analyzer mid-stream, or hit Gemini without `GEMINI_API_KEY`) → stream emits `{ kind: 'error', code, message, detail }`; UI shows the message inline with an expandable "Details" block; user can click "Start fresh" to retry.
5. **Click "Start fresh"** from the re-parse dialog → request body includes `{ fresh: true }`; server discards cached partials and re-runs from phase 1.
6. **Pick a model** from the re-parse dialog (e.g. `gemma-3-27b`) → request body includes `{ model: 'gemma-3-27b' }`; subsequent analysis uses that model. `ui.selectedModel` updates.
7. **Cancel mid-stream** (navigate away) → stream connection closes; no further ticks. Returning to `#/books/:bookId/analysing` resumes a fresh stream (server decides whether to resume from cache or restart).
8. **Empty result** — if the analyzer returns zero characters or zero chapters, the stream still emits a `result` event; frontend transitions to `confirm` with an empty cast and the user sees an empty state in the confirm view (no crash).
9. **Per-chapter failure → paused, awaiting retry** — when a Gemini 5xx (or analyzer JSON repair miss) trips a single chapter's cast detection, the stream continues past that chapter, records it in `cache.failedChapterIds`, and emits `chapter-failed` for the UI. At the end of Phase 0a, if any chapter is still failed, the route emits `{ kind: 'error', code: 'cast_incomplete' }` and ends without advancing to Phase 1. The failed-chapter panel renders with "Paused — N chapter(s) still need cast detection" and active Retry buttons. Clicking Retry hits the subset endpoint; on success the row disappears. Once every failed chapter resolves the view auto-fires a fresh `/analysis/stream` that advances to Phase 1 — the user never has to click "Try again" themselves.
10. **Retry mid-stream pauses, runs subset, resumes** — failed-chapter rows can appear while Phase 0 is still streaming other chapters (or hydrate from book-state for prior-run failures). Click Retry while the main run is in flight: the SSE aborts, the subset call runs alone, then the main run re-fires automatically on settle. The subset endpoint's `chapter-failed` re-emission upserts the row (with the fresh error message); a clean success drops the row via `chapter-resolved`. When the main run's Phase 0a re-queue succeeds for a row that was hydrated from a prior `failedChapterIds`, the same `chapter-resolved` event clears the row mid-stream so the user doesn't click Retry on something the server has already fixed.
11. **Dropped-quote panel** — after the verify pass at the end of Phase 0, the panel under the cast preview surfaces any quotes the verifier rejected (model fabrications). Two reasons render: "not in source" (stitched dialogue or hallucinated lines) and "empty after normalisation" (pure punctuation / whitespace). PowerShell audit: `Get-Content .audiobook/dropped-quotes.json | ConvertFrom-Json`. Grep into `.batches[-1].entries` for the latest pass; the array grows on every run.

## Recovering from a corrupted cast cache

For books whose `.audiobook/` files drifted out of sync with the analysis cache before the A1/A2/A3 guards landed (orphan `characterId` references in `manuscript-edits.json`, `cast.json` collapsed to a tiny roster while the cache still has Phase 1 sentences attributing to characters that no longer exist), wipe the per-book state with the reconciliation script:

```powershell
npm run reconcile-cast -- -BookId shannon-messenger__the-hollow-tide__unlocked
```

The script (driven by `scripts/lib/cast-reconcile.psm1` so the archive/remove logic is Pester-tested) walks the workspace tree for the matching `state.json`, prints a summary of the current cast/edits/cache state (with any orphan IDs highlighted), prompts to confirm, then archives `cast.json`, `manuscript-edits.json`, and the analysis-cache entry to `.audiobook/.reconcile-backup-<timestamp>/`. `state.json`, `change-log.json`, and `dropped-quotes.json` are left in place so the next analysis run picks up where it left off.

After running, re-open the book and click Start analysis to repopulate the cast.

Workspace resolution mirrors the server's: `server/user-settings.json` `workspaceDirOverride` > `server/.env` `WORKSPACE_DIR` > `<repo>/audiobook-workspace`. Pass `-WorkspaceDir <path>` to override.

## Out of scope

- Specific phase timings or labels — those are server-driven.
- Resumability of partial progress — depends on analyzer mode (covered in `05-analyzer-manual-handoff.md` and `06-analyzer-gemini.md`).
- Network reconnection — stream errors are surfaced, not silently retried.
