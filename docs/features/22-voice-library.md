# Voice library

> Status: stable
> Key files: `src/views/voices.tsx`, `src/store/voices-slice.ts` (`hydrate`, `setPinned`, `hydrateBaseVoices`, `setOverride`), `src/lib/api.ts` (`getVoices`, `setVoicePin`, `getBaseVoices`, `setVoiceOverride`), `src/lib/play-sample-with-auto-load.ts` (`playBaseVoiceSampleWithAutoLoad`), `server/src/routes/voices.ts`, `server/src/tts/base-voices.ts`, `src/lib/voice-palette.ts`, `server/src/tts/voice-palette.ts`
> URL surface: `#/voices`
> OpenAPI ops: `GET /api/voices`, `GET /api/voices/base`, `PUT /api/voices/:id/pin`, `PUT /api/voices/:voiceId/override`

## What this covers

Cross-book view of every voice the user has confirmed. Cast members are grouped by the resolved model speaker (their "voice family" — engine + speaker name) so all characters that sound the same gather under one header, with a separate `Base voices` tab for auditioning each engine's raw speaker catalog. Supports pinning, per-family Play sampling, and manual base-voice overrides written through to every `cast.json` that shares the same `voiceId`. Used both as a browsable surface and as a source for cross-book voice reuse during cast confirmation.

## Invariants to preserve

- `GET /api/voices?currentBookId=X&engine=Y` returns `{ voices: Voice[] }`; both query params are optional but, when present, narrow the result (`src/lib/api.ts:299-307`).
- `GET /api/voices/base` returns `{ voices: BaseVoice[] }` — the raw per-engine speaker catalog (Coqui live from sidecar `/speakers` with static fallback when the sidecar is down; all 30 published Gemini voices; Piper/Kokoro intentionally empty until their voice-mapping tables land). In-process cached with a 5 min TTL; the cache is invalidated whenever an override PUT updates at least one `cast.json`. Implementation: `server/src/tts/base-voices.ts` (`listBaseVoices`, `invalidateBaseVoiceCache`).
- `PUT /api/voices/:id/pin { pinned: boolean }` updates the pin state server-side; response is empty 2xx.
- `PUT /api/voices/:voiceId/override` body shape: `{ override: { engine, name } }` to set, `{ override: null }` to clear. Walks every confirmed-cast book and writes (or deletes) `overrideTtsVoice` on every character whose `voiceId` matches — the same voiceId recurring across a series stays in sync (`server/src/routes/voices.ts:263-331`, `applyOverrideToCastFiles`). 204 on success, 404 when no character matches, 400 on malformed body.
- `Voice.gradient` is `[string, string]` tuple (`src/lib/types.ts:37-39`) — explicit override of OpenAPI's `string[]` widening. Renderer assumes tuple shape.
- Voice record: `{ id, character, bookTitle, bookId, bookSeries: string | null, attributes, gradient, usedIn, source: 'current' | 'library', ttsVoice, overrideTtsVoice?: { engine, name } | null }`. `source: 'current'` means "this book"; `'library'` means "another book." `bookSeries` is parsed from the `books/<Author>/<Series>/<Book>` layout (`server/src/routes/voices.ts:60`); the synthetic series name `"Standalones"` is normalised to `null` so standalones flatten under the family header rather than getting a fake series sub-header (`server/src/routes/voices.ts:110-117`).
- `ttsVoice` already reflects the override when the override engine matches the `engine` query param; `overrideTtsVoice` is surfaced separately so the UI can render Manual / Engine-mismatch badges without re-deriving the resolution. When two books share a `voiceId` and one carries an `overrideTtsVoice`, the first non-null override the aggregator sees wins (`server/src/routes/voices.ts:158-160`); conflicting overrides are not expected because the PUT walks every `cast.json` atomically.
- Pinning is optimistic in the slice: dispatch `setPinned(id, true)` → UI updates immediately; PUT fires in background; on error, slice rolls back and shows a toast.
- The same voice may appear with both `source: 'current'` and `source: 'library'` if it's been used across books; the renderer must dedupe by `id`.
- **Voices view groups cast by voice family**, where a family is the resolved `(engine, ttsVoice.name)` pair (`src/views/voices.tsx:24-38` — `VoiceFamily`; `:85` — `buildFamilies` `useMemo`). Each family renders as one header gathering every cast member that maps to the same speaker, with nested groups `seriesGroups: SeriesGroup[] → books: BookGroup[] → voices: Voice[]`. The pre-commit "one `<section>` per book" grouping is gone. Families sort pinned-first; series sort under each family; books under each series; characters under each book.
- Four tabs: `all | current | library | base` (`Tab` type at `src/views/voices.tsx:20`). `all` shows every family. `current` filters families to members with `source === 'current'`. `library` filters to `source === 'library'`. `base` swaps the family list for the `BaseVoiceCatalogPanel`, driven by `state.voices.baseVoices` (hydrated lazily on mount via `api.getBaseVoices()`).
- **Sample playback goes through `playBaseVoiceSampleWithAutoLoad`** (`src/lib/play-sample-with-auto-load.ts`) for both per-family Play (`src/views/voices.tsx:97-115`) and per-base-voice Play in the Base voices tab (`:119-138`). Same JIT contract as plan 10 — evict analyzer → load sidecar → synthesize, with `onStatus` cycling `evicting → loading-tts → synthesizing` shown as a per-row status pill keyed by `family.key` or `${engine}|${name}`. The helper caches by `(engine, speakerName)` so unused speakers don't fragment the on-disk cache across the voiceId paths that triggered them.
- The inline pin (star) lives on the voice card itself, next to the character name + reuse pill (`src/components/voice-library-panel.tsx::VoiceCard` `onTogglePin` prop). The sidebar `VoiceLibraryPanel` (cast view) does **not** render the pin.
- **Voice swatch colour is derived from the resolved prebuilt TTS voice** (`ttsVoice.name`), not from a hash of the voice id. Two characters that route to the same prebuilt (e.g. both on `Viktor Menelaos`) get the same swatch gradient — at-a-glance signal that they will sound the same. The palette is 16 gradients arranged as 8 profile-bucket families × 2 slots (`male-deep`, `male-mid`, …, `narrator-cool`) in `BUCKET_GRADIENTS`. Coqui slot N and Gemini slot N within the same bucket share a gradient, so flipping the engine doesn't reshuffle the cast view. Unknown/custom voice names fall back to a stable hash of the voice id. The server stamps `Voice.gradient` at scan time (`server/src/tts/voice-palette.ts:gradientForTtsVoice`); the frontend mirrors the table at `src/lib/voice-palette.ts` for stub voices the cast view and profile drawer build before any library record exists. The two files MUST agree on per-voice gradient — if they drift, the swatch will jump when stub data flips to server-derived data. Both sides are covered by exhaustive Vitest specs that walk every entry in `COQUI_PROFILE_VOICES` / `GEMINI_PROFILE_VOICES`.
- Character-avatar colour (`Character.color` → `CHAR_COLORS` in `src/lib/colors.ts`) is a separate axis: 30 slot colours allocated server-side by roster order during analysis. The avatar identifies the *character*; the swatch identifies the *voice*. Don't conflate the two palettes.

## Acceptance walkthrough

Run `VITE_USE_MOCKS=true`.

1. **Open `#/voices`** → on the default `All` tab, the list renders one header per voice family (engine + speaker name), pinned families first. Standalone books flatten under the family header (no "Standalones" series sub-header). Cast members nest series → book → character.
2. **Click the per-family Play button** → status pill on the header cycles `Preparing… → Freeing memory… → Loading TTS… → Synthesising…` (same JIT contract as plan 10), then audio plays inline. A second click within the cache TTL returns cached audio without the load cycle.
3. **Toggle pin on a voice** → card moves to the pinned section of its family instantly (optimistic); `PUT /api/voices/<id>/pin` fires in background; on success, no further UI change; on failure, card rolls back and a toast appears. The family's `anyPinned` flag flips and the family bubbles to the top of the list.
4. **Switch to the `Current` / `Library` tabs** → list narrows to families whose members have the matching `source`. Switching to `Base voices` → list swaps to the raw catalog: one row per engine speaker (Coqui live + 30 Gemini), each with a Play button and a "Used by N cast members" chip.
5. **Click Play on a base-voice row** → same JIT status pill cycle as per-family Play. Audio plays inline.
6. **From a book context** (open `#/voices?currentBookId=<id>` via the cast view's "Browse library" CTA) → families containing `source: 'current'` voices for that book bubble up.
7. **Click a voice card** → opens the profile drawer for the linked character (in the source book) — this requires loading that book; or alternatively opens a read-only preview drawer. Either UX is acceptable; document the chosen path.
8. **Override walkthrough** — from the profile drawer (per plan 10), pick a different Model voice; close the drawer and reload `#/voices`. The character now appears under the new family header. If the override engine differs from the project's active engine, the family rendering shows the Engine-mismatch state inherited from the drawer (override persists; synth still uses the auto-resolved voice).
9. **Real-mode regression** — switch to `VITE_USE_MOCKS=false`. Library populates from disk scan; pinning and overrides persist across reloads; the Base voices Coqui list is sourced from the live sidecar `/speakers` (falls back to the static catalog if the sidecar is down).

## Out of scope

- Voice creation from scratch (not in v1).
- Bulk pinning / bulk delete.
- Voice "remixing" (blend two voices into a new one).
