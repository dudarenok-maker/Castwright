# 166 — GitHub Issues for backlog detail + bug tracking

> Status: active — tooling + conventions + docs land in PR 1 (this plan); the bulk migration + BACKLOG.md thinning run as PRs 2–3 against the live repo.
> Key files: `scripts/{gh-labels,migrate-backlog-to-issues,thin-backlog}.mjs`, `.github/ISSUE_TEMPLATE/{backlog-item,bug,config}.yml`, `docs/BACKLOG.md`, `CONTRIBUTING.md` (§Issues), `CLAUDE.md` (§The backlog).
> URL surface: none (process / tooling / docs).
> OpenAPI ops: none.

## Benefit / Rationale

- **User (maintainer):** backlog items gain triage, discussion threads, and PR-linking; bugs get a tracked home (history, severity, "Closes #NN" on the fixing PR) instead of living only in conversation. `docs/BACKLOG.md` stops carrying the full spec of every item and becomes a fast prioritized scan.
- **Technical:** detail lives in exactly one place (the issue), ending the dual-maintenance risk of keeping What/Acceptance/Key-files in both a markdown list and anywhere else. The migration is idempotent and the thinning is reversible (git-tracked), so the cutover is safe to re-run / roll back.
- **Architectural:** the `<prefix>-<n>` ID stays the durable cross-reference (code/commits/plans don't rot), now mapped 1:1 to a GitHub issue whose `#NN` is the auto-close hook. The label taxonomy + issue forms are version-controlled, so they reproduce on a fresh clone or when the repo goes public.

## Architectural impact

- **New seams / extension points:**
  - `scripts/gh-labels.mjs` — the label taxonomy as code (exported `LABELS`), upserted via `gh label create --force`.
  - `scripts/migrate-backlog-to-issues.mjs` — parses `docs/BACKLOG.md` and files one issue per item; exports the pure helpers `parseBacklogItems`, `issueIdFromTitle`, `issuePayload`.
  - `scripts/thin-backlog.mjs` — one-shot rewrite of `docs/BACKLOG.md` to the thin planning view, driven by `docs/backlog-issue-map.json`.
  - `.github/ISSUE_TEMPLATE/` — YAML issue forms (`backlog-item`, `bug`) + `config.yml` (`blank_issues_enabled: false`).
- **Invariants preserved:** `<prefix>-<n>` IDs remain permanent and never reused (the migration keys on the ID token, not wording); MoSCoW buckets + positional priority survive the thinning; **bugs stay off `docs/BACKLOG.md`** (still out-of-band); the merge-commit-only policy makes `Closes #NN` in the PR body the auto-close path.
- **Migration story:** `docs/backlog-issue-map.json` (`{ "<prefix>-<n>": <issueNumber> }`) is the single artifact bridging the two systems. The migration is **idempotent** — it lists existing issues once and keys them by the `<prefix>-<n>` token parsed from each title, so re-running creates nothing new. The thinning is a separate **one-shot** transform (not idempotent — kept out of the migration script so the migration stays re-runnable).
- **Reversibility:** `docs/BACKLOG.md` is git-tracked (`git checkout docs/BACKLOG.md` reverts the thinning); created issues can be bulk-closed; the map file can be regenerated by re-running the (idempotent) migration.

## Invariants to preserve

- `parseBacklogItems` only treats a `### `/`#### ` heading whose text **leads with** a backtick-wrapped `(fe|srv|side|ops|fs)-\d+` token as an item — sub-group titles (e.g. `### Audio & playback`) and ID-less shipped-note paragraphs (`_`fs-2` … shipped …`) are skipped (`scripts/migrate-backlog-to-issues.mjs`, asserted in `scripts/tests/migrate-backlog-to-issues.test.mjs`).
- A struck-through / `RESOLVED` entry (e.g. `srv-17`) is **reported, not filed** — `isResolved` triggers on the `RESOLVED` token or a title that *starts* with `~~`, so inline strikethrough mid-title (`srv-4`'s `~~jsdom~~`) does **not** misclassify it.
- `issueIdFromTitle(issuePayload(item).title) === item.id` for every parsed item — the idempotency round-trip (unit-tested), so a re-worded "what" can never produce a duplicate issue.
- The label taxonomy in `gh-labels.mjs` `LABELS` is the source of truth: 5 `area:` + 4 `moscow:` + 3 `type:`/`bug` + 2 helpers (14 total).

## Test plan

### Automated coverage

- node:test (`scripts/tests/migrate-backlog-to-issues.test.mjs`) — runs via `npm run test:hooks` (in `verify`). Asserts: item extraction with correct tier/prefix; sub-group titles + the Retired-numbering section are not items; struck-through `RESOLVED` entries are reported-not-filed; inline `~~strikethrough~~` doesn't mark an item resolved; the tracking heuristic (`type:chore`+`tracking` vs `type:feature`); the shipped-note paragraph doesn't bleed into the preceding item's body; and the `issueIdFromTitle ∘ issuePayload` round-trip (idempotency key).

`npm run test:scripts` (Pester) covers `scripts/lib/` PowerShell helpers only — it does **not** exercise these Node scripts. The pure-helper unit test above is the real regression lock; the side-effecting `gh` glue is verified by the manual dry-run/apply walkthrough (the scripts are dry-run-by-default).

### Manual acceptance walkthrough

1. `node scripts/gh-labels.mjs` → prints 14 `gh label create … --force` commands; `--apply` → `gh label list` shows all 14.
2. `node scripts/migrate-backlog-to-issues.mjs` → prints the ~57-item manifest with labels + the `srv-17` skip line; **`--apply`** → `gh issue list` shows them all labelled, and `docs/backlog-issue-map.json` is written. **Re-run `--apply`** → "created 0, reused N" (idempotent).
3. `node scripts/thin-backlog.mjs` (after step 2 wrote the map) → prints a unified diff collapsing each item to the thin form; `--apply` → `docs/BACKLOG.md` is thin with working `#NN` links; `git diff` to review, `git checkout` to revert.
4. File a test bug via the **Bug** form → it lands with the `bug` label, off `docs/BACKLOG.md`; a Backlog item via the **Backlog item** form → lands with `type:feature`.

## Out of scope

- A GitHub Projects (v2) board — labels + `docs/BACKLOG.md` are the planning surface by decision (less overhead, single source of priority).
- Migrating away from the `<prefix>-<n>` scheme to issue numbers — the prefix ID stays canonical.
- Server-side label/field enforcement (Free plan, no Actions budget) — the forms + `blank_issues_enabled: false` are the soft gate.
- Auto-syncing `docs/BACKLOG.md` ⇄ issues on an ongoing basis — the thinning is a one-shot cutover; from then on humans keep the thin row + issue in step per the CONTRIBUTING.md "Issues" convention.
