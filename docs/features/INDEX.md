# Feature regression plans

Living specs for every feature shipped in v1. Each plan combines **invariants to preserve** (structural rules a refactor must not break) and an **acceptance walkthrough** (manual click-through with expected URL hashes, redux state, and network calls). Partial features are documented as-is with a `KNOWN: scaffolded` banner so reviewers do not mistake "documented current behavior" for "fully working."

PRs that change behavior cited in a plan MUST update the relevant plan in the same diff ‚Äî that is the regression discipline the plans buy.

**For outstanding work** see [`../BACKLOG.md`](../BACKLOG.md) ‚Äî the MoSCoW-bucketed list of every follow-up, scaffolded item, and untested seam pulled from these plans. Future planned rounds of work pull from there.

## Writing a new plan

1. Copy [TEMPLATE.md](TEMPLATE.md) ‚Äî required sections are **Benefit / Rationale**, **Architectural impact**, and **Test plan** (automated + manual).
2. Add YAML frontmatter (`status`, `shipped`, `owner`). New plans start `status: draft` until they ship.
3. Link the plan from the matching area section below.
4. Land paired automated tests in the same PR ‚Äî see CLAUDE.md "Testing discipline".

## Plan lifecycle

Each plan's frontmatter `status:` is one of:

- **draft** ‚Äî written, not yet implementing.
- **active** ‚Äî implementation in progress on the main branch.
- **stable** ‚Äî feature is end-to-end functional, behavior locked by automated tests.
- **deferred** ‚Äî explicitly parked; see body for "wake when" condition.
- **scaffolded** ‚Äî UI/contract in place, parts mocked or partial (matches the `KNOWN: scaffolded` body banner).

When a plan reaches **stable** AND has a filled **Ship notes** section, move it to [`archive/`](archive/README.md) in the same PR ‚Äî the top-level index is the working set, not a changelog. See `archive/README.md` for the move checklist.

## How to run a plan

1. Pick a plan from the index.
2. Read the **Invariants** section first; if any cited file/line has moved, the plan is stale ‚Äî fix it before validating behavior.
3. Walk the **Acceptance** steps against a running app (`npm run dev`, plus `cd server && npm run dev` and the TTS sidecar if the plan covers audio).
4. Any drift ‚Üí update the plan in the same PR as the code change.

## Status legend

In-body banner (legacy ‚Äî still valid for plans that haven't been ported to frontmatter):

- **stable** ‚Äî feature is end-to-end functional; assert real behavior.
- **KNOWN: scaffolded** ‚Äî UI/contract is in place but parts are mocked or partial; assert only the documented behavior.
- **KNOWN: backend-pending** ‚Äî frontend done, backend stub returns empty/canned data; mock mode exercises the UI, real mode is intentionally a no-op.
- **KNOWN: operational dependency** ‚Äî works but requires a sibling process the user must start (TTS sidecar).
