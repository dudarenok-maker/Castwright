# torch + torchaudio — EXPLICIT, matched, pinned. Qwen synth needs torch (~10
# import sites in main.py); torchaudio is tightly version-coupled to torch.
# (coqui-tts 0.27.5 declares torch only under its [cuda]/[cpu] extras, so an
# explicit line is required — see test_requirements.py.) Pinned to 2.11.0:
# clears CVE-2025-2999 (unpack_sequence) + CVE-2025-3001 (lstm_cell); highest
# line with a cu128 wheel (2.12 is cu130-only → driver bump). torchaudio 2.9
# removed the in-core soundfile backend (load/save now need torchcodec) — this
# is fine because the sidecar NEVER calls torchaudio.load: audio I/O is
# soundfile + ffmpeg, and Coqui uses manifest speakers, not speaker_wav.
#
# These plain pins do NOT carry a CUDA build by themselves: PyPI's default torch
# wheel is CPU-only on Windows. bootstrap-venv pre-installs torch/torchaudio from
# the cu128 index (installRecipe('nvidia').torchPreinstall) BEFORE this overlay,
# so the +cu128 build is already present and these `==2.11.0` lines just confirm
# it. To pin a specific CUDA build by hand, pre-install before the requirements:
#   pip install torch==2.11.0 torchaudio==2.11.0 --index-url https://download.pytorch.org/whl/cu128
torch==2.11.0
torchaudio==2.11.0

# NOTE on DeepSpeed: not in requirements because its setup.py needs the
# standalone NVIDIA CUDA Toolkit (with nvcc) at install time — the CUDA
# runtime libraries that ship inside the PyTorch wheel aren't enough. On
# Windows boxes without the toolkit, `pip install deepspeed` fails at
# metadata-generation. If you have (or are willing to install) the CUDA
# Toolkit from https://developer.nvidia.com/cuda-downloads, then:
#   DS_BUILD_OPS=0 ./.venv/Scripts/python.exe -m pip install --no-build-isolation "deepspeed>=0.15,<0.16"
# and set COQUI_DEEPSPEED=1 in server/.env. The sidecar already handles
# both states — it logs a warning and continues in vanilla mode if
# DeepSpeed isn't available.

# Kokoro v1 — quality-tuned local TTS engine, eagerly loaded at sidecar startup.
# Installed PLAIN (no [gpu] extra) so it deterministically lands the core
# `onnxruntime` (CPU) module; install-ort.mjs then swaps in the GPU runtime the
# profile needs (nvidia → onnxruntime-gpu) as the single enforcement point. We do
# NOT use kokoro-onnx[gpu]: that extra coexists with the core onnxruntime dep, and
# pip's resolution order can leave the CPU build owning the shared `onnxruntime/`
# namespace → a silent CPU-only Kokoro on a GPU box (the 2026-06-16 regression).
# macOS/Apple-Silicon keeps the plain CPU/CoreML runtime (no swap for non-nvidia).

# into voices/kokoro/ (gitignored, ~330 MB).
